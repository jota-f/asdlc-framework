# Plano: Melhorias do A-SDLC baseadas no Workshop de Matt Pocock

## Contexto

Análise do vídeo "How I Use AI to Code (Workshop)" de Matt Pocock revelou 5 melhorias propostas para o framework A-SDLC. Após análise da codebase, identifiquei gaps reais e priorizei as melhorias por impacto e viabilidade.

## Melhorias Priorizadas

### Melhoria 1: Agente Questionador ("Grill Me") — ALTO IMPACTO
**Gap:** O Requirements Agent atual é passivo — documenta requisitos mas não os desafia. A workflow `create_asdlc_story.md` apenas transforma demanda em story sem validar completude.

**Implementação:**

#### 1.1 Nova Workflow: `agentic_templates/workflows/grill_requirements.md`
- Comando `/asdlc-grill` que ativa um ciclo de questionamento antes da criação da story
- O agente deve fazer 5-8 perguntas obrigatórias sobre: ator, contexto, edge cases, critérios de sucesso, restrições não-mentionadas, dependências implícitas
- Workflow termina com um "Shared Understanding Document" que alimenta a story

#### 1.2 Atualizar `.asdlc/agents/requirements_agent.md`
- Adicionar seção "Modo Grill" com instruções de questionamento proativo
- Adicionar checklist de completude: ator definido? ação clara? benefício mensurável? edge cases listados?

#### 1.3 Atualizar `agentic_templates/workflows/create_asdlc_story.md`
- Adicionar passo opcional: "Se a demanda for vaga, sugira `/asdlc-grill` antes de prosseguir"
- Adicionar validação: se a story não tiver user story completa (ator + ação + benefício), recuse

#### 1.4 Atualizar `agentic_templates/skills/asdlc_story_generator/SKILL.md`
- Adicionar seção "Validação de Qualidade de Entrada"
- Verificar se a demanda tem especificidade suficiente antes de gerar a story

### Melhoria 2: TDD Obrigatório no Pipeline — ALTO IMPACTO
**Gap:** O pipeline atual faz Code → Test (sequencial). O Test Agent cria testes DEPOIS do código, permitindo que o Code Agent "trapaceie" escrevendo código que passa em testes superficiais.

**Implementação:**

#### 2.1 Atualizar `agentic_templates/skills/asdlc_implementation/SKILL.md`
- Modificar Passo 3 (Garantia da Qualidade) para exigir TDD:
  - 3.1: LEIA os Critérios de Aceitação
  - 3.2: CRIE testes que FALHEM primeiro (Red)
  - 3.3: IMPLEMENTE o código até os testes passarem (Green)
  - 3.4: REFATORÉ se necessário (Refactor)
- Adicionar regra: "NUNCA implemente código funcional sem antes ter um teste que falhe para o cenário"

#### 2.2 Atualizar `asdlc/story_manager.py` (`_executar_etapas_implementacao`)
- Inverter ordem: Test Agent deve criar testes ANTES do Code Agent (Fase 2 → Fase 1)
- Nova sequência: Architecture → **Test (Red)** → Code (Green) → Review
- Adicionar validação: após Test Agent criar testes, rodar `run_command` e confirmar que testes FALHAM

#### 2.3 Atualizar `asdlc/agent_executor.py`
- Adicionar parâmetro `tdd_mode: bool = False` em `validate_and_fix`
- Quando `tdd_mode=True`: primeiro spawn do Test Agent para criar testes, depois spawn do Code Agent

#### 2.4 Atualizar `.asdlc/agents/test_agent.md`
- Adicionar seção "Modo TDD (Red Phase)"
- Instruções: "Crie testes que descrevam o comportamento esperado. Os testes DEVEM falhar neste ponto."

### Melhoria 3: Fatias Verticais ("Tracer Bullets") — MÉDIO IMPACTO
**Gap:** O `plan_generator.py` gera tarefas horizontais (ex: "Criar modelo", "Criar API", "Criar UI"). O A-SDLC não orienta o arquiteto a priorizar o caminho funcional mínimo.

**Implementação:**

#### 3.1 Atualizar `asdlc/plan_generator.py`
- Adicionar no prompt do arquiteto: "Priorize TRACER BULLETS — fatias verticais que atravessam todas as camadas. Em vez de criar todos os modelos primeiro, crie o mínimo caminho funcional de ponta a ponta."
- Adicionar exemplo de tracer bullet no prompt:
  ```
  **Exemplo de Tracer Bullet (CORRETO):**
  Tarefa 1: Criar modelo User + endpoint POST /users + teste de integração
  Tarefa 2: Criar modelo Post + endpoint GET /posts + teste de integração

  **Exemplo de Horizontal (EVITAR):**
  Tarefa 1: Criar todos os modelos (User, Post, Comment)
  Tarefa 2: Criar todos os endpoints
  Tarefa 3: Criar todos os testes
  ```

#### 3.2 Atualizar `agentic_templates/skills/asdlc_story_generator/SKILL.md`
- Adicionar orientação: "Quando possível, organize tarefas como fatias verticais (tracer bullets) — cada tarefa deve ser funcional e testável independentemente"

#### 3.3 Atualizar `agentic_templates/workflows/scope_analysis.md`
- Na seção "Decisão de Quebra", adicionar critério: "Prefira fatias verticais a camadas horizontais"

---

### Melhoria 4: Monitoramento de Densidade de Contexto — MÉDIO IMPACTO
**Gap:** O Context Compactor é acionado por contagem de mensagens (30+), não por densidade de tokens. O `agent_executor.py` trunca PROJECT_CONTEXT em 3000 chars e arquivos em 5000 chars, mas não monitora o contexto total do agente.

**Implementação:**

#### 4.1 Atualizar `asdlc/agent_executor.py`
- No `prepare_context()`, adicionar contagem de tokens estimada (chars / 4 ≈ tokens)
- Adicionar log warning se contexto estimado > 80k tokens
- Adicionar log error se contexto estimado > 100k tokens ("DUMB ZONE DETECTED")
- Retornar metadata com `context_token_estimate` junto com o prompt

#### 4.2 Atualizar `asdlc/story_manager.py`
- No `_executar_etapas_implementacao`, após cada fase, verificar se o contexto cresceu demais
- Se > 80k tokens: invocar automaticamente o Context Compactor antes da próxima fase

#### 4.3 Atualizar `agentic_templates/skills/asdlc_context_compactor/SKILL.md`
- Adicionar trigger automático: "Se o contexto estimado ultrapassar 80k tokens, compacte imediatamente"
- Adicionar seção "Smart Zone" com limites recomendados

#### 4.4 Atualizar `agentic_templates/skills/asdlc_implementation/SKILL.md`
- Adicionar no Passo 1: "Monitore o tamanho do contexto. Se ultrapassar 80k tokens estimados, invoque `asdlc_context_compactor` antes de continuar"

### Melhoria 5: Revisão de Profundidade de Módulos — BAIXO IMPACTO (fase 2)
**Gap:** O Architecture Agent não avalia se módulos são "profundos" (interface simples, lógica complexa) vs "rasos" (muitos módulos com interfaces espalhadas).

**Implementação (simplificada):**

#### 5.1 Atualizar `.asdlc/agents/architecture_agent.md`
- Adicionar seção "Análise de Profundidade de Módulos"
- Critérios: "Prefira módulos profundos — interfaces simples que escondem complexidade. Evite módulos rasos onde cada função expõe detalhes internos."
- Adicionar ao checklist: "[ ] Módulos têm interfaces simples e coesas"

#### 5.2 Atualizar `asdlc/plan_generator.py`
- No prompt do arquiteto, adicionar: "Avalie se os módulos propostos são profundos (interface simples, lógica encapsulada) ou rasos. Se identificar módulos rasos, sugira consolidação."
